# 阶段 5：质量检查

## 职责

验证生成数据的正确性、完整性和一致性。

## 输入

- `/tmp/dvp_raw_materials.json`（阶段 1 的输出，用于外部正确性比对）
- `/tmp/dvp_data.json`（阶段 4 的输出）

## 做法

### 1. 内部逻辑检查（R1-R6）

#### R1: 字段引用完整性

**检查内容**：编辑检查（Edit Checks）中引用的变量名是否在 CRF 字段列表中存在。

**判断逻辑**：
1. 提取所有编辑检查章节中 `变量` 或 `变量名` 列的值
2. 与 CRF 相关章节中的字段列表对比
3. 编辑检查中引用的变量不在 CRF 中 → ❌ 错误
4. CRF 中有字段但无对应编辑检查 → ⚠️ 警告（可能遗漏）

**触发条件**：DVP 中同时包含编辑检查章节和 CRF/数据域章节时执行。

#### R2: 数据一致性

**检查内容**：同一条信息在不同章节中的值是否一致。

**判断逻辑**：
1. 提取 meta 区域和各章节中出现的共同字段（如研究名称、方案编号、访视名称）
2. 比较同一字段在不同位置出现的值
3. 值不一致 → ❌ 错误
4. 某处缺失但其他处存在 → ⚠️ 警告

**典型检查项**：
- 研究名称在 meta 和章节中是否一致
- 方案编号在 meta 和章节中是否一致
- 访视名称在不同章节间是否一致

#### R3: 命名规范

**检查内容**：ID 编号的连续性和格式统一性。

**判断逻辑**：
1. 提取所有带 ID 前缀的标识符（如 EC001、CC001、VS001 等）
2. 按前缀分组检查：
   - 同类 ID 的前缀是否一致 → 不一致 ⚠️ 警告
   - 编号是否连续 → 不连续 ⚠️ 警告
   - 是否有重复 ID → ❌ 错误
3. 格式检查（如是否统一使用 3 位数字编号）

#### R4: 业务逻辑合理性

**检查内容**：日期范围、数值区间、逻辑条件是否合理。

**判断逻辑**：
1. **日期检查**（根据字段语义模式匹配，不依赖固定字段名）：
   - 出生日期不能晚于研究日期 → ❌ 错误
   - 知情同意日期不能晚于首次给药日期 → ❌ 错误
   - 访视日期应在研究周期内 → ⚠️ 警告

2. **数值区间检查**：
   - 正常范围下限应小于上限 → ❌ 错误
   - 数值约束条件是否在合理范围内 → ⚠️ 警告

3. **逻辑条件检查**：
   - 条件表达式语法是否合理（如 `>` `<` `=` `!=` `in` `notin` 等操作符使用是否正确）
   - 空值条件 `!= ""` 和 `== ""` 是否矛盾 → ❌ 错误

#### R5: 规则冲突检测

**检查内容**：同一字段是否存在矛盾的检查规则。

**判断逻辑**：
1. 按变量名分组所有编辑检查
2. 对同一变量的多条规则对比：
   - 一条要求必填（`!= ""`），另一条允许为空 → ❌ 错误
   - 一条设置上限，另一条设置的上限低于下限 → ❌ 错误
   - 两条规则检查相同条件但严重程度不同 → ⚠️ 警告

#### R6: 交叉引用一致性

**检查内容**：章节间的 ID 引用是否对应。

**判断逻辑**：
1. 提取所有章节中出现的 ID 引用（如"参见 EC001"、"详见 CC003"等）
2. 验证被引用的 ID 是否存在
3. 引用的 ID 不存在 → ❌ 错误
4. 存在未在任何地方被引用的 ID → ⚠️ 警告

### 2. 外部正确性检查（R7-R9）

#### R7: 原文一致性检查

**检查内容**：生成内容与 Protocol/CRF/DMP 原文的关键信息是否一致。

**判断逻辑**：
1. 提取 DVP 中由文档预填充的关键字段值（研究名称、方案编号、访视名称等）
2. 与 `/tmp/dvp_raw_materials.json` 中的原文提取结果对比
3. 值不一致 → ❌ 错误
4. 原文存在但 DVP 中缺失 → ⚠️ 警告

**触发条件**：存在原料数据可交叉比对时执行。

#### R8: 覆盖完整性

**检查内容**：CRF 中的关键字段是否都有对应的编辑检查或验证规则。

**判断逻辑**：
1. 从原料数据中提取 CRF 的全部字段列表
2. 从 DVP 编辑检查章节中提取被覆盖的变量列表
3. CRF 字段无对应编辑检查 → ⚠️ 警告（可能遗漏）
4. 数据收集类字段（如 AE、ConMed）完全无覆盖 → ❌ 错误

**触发条件**：DVP 同时包含 CRF 数据和编辑检查章节时执行。

#### R9: 业务合规性

**检查内容**：编辑检查的业务逻辑是否与文档中的业务要求一致。

**判断逻辑**：
1. 从 Protocol 中提取明确的业务规则（如"知情同意日期必须早于首次给药"）
2. 检查 DVP 编辑检查中是否已体现这些规则
3. 明确的业务规则未体现 → ❌ 错误
4. 编辑检查的条件与文档描述不符 → ❌ 错误
5. 文档建议但不强制的规则未体现 → ⚠️ 警告

**触发条件**：存在 Protocol 原文且 DVP 包含编辑检查章节时执行。

### 3. 输出检查报告

```
=== DVP 质量检查报告 ===

内部逻辑检查：
  R1 字段引用完整性    ✅ 通过
  R2 数据一致性        ❌ 访视名称不一致："Screening" vs "筛选访视"
  R3 命名规范          ⚠️ EC003 后跳到 EC005
  R4 业务逻辑合理性     ✅ 通过
  R5 规则冲突检测       ✅ 通过
  R6 交叉引用一致性     ✅ 通过

外部正确性检查：
  R7 原文一致性        ✅ Protocol 关键信息一致
  R8 覆盖完整性        ⚠️ CRF 中 ETHNIC 字段无对应编辑检查
  R9 业务合规性        ✅ DMP 流程描述一致

---
总计：✅ 8  ⚠️ 2  ❌ 1
❌ 错误项必须修正，⚠️ 警告项由用户决定。
```

### 4. 用户决定

| 级别 | 处理 |
|------|------|
| ❌ 错误 | 必须修正（可当前阶段修正或回退至阶段 4 重新生成） |
| ⚠️ 警告 | 用户决定是否修正，可说"接受"跳过 |

用户补充新信息时，回到阶段 1。用户纠正思路时，在当前阶段修正。

## 输出

检查报告直接展示，不写文件。如需修正，更新 `/tmp/dvp_data.json`。

## 完成条件

- 无 ❌ 错误项
- ⚠️ 警告项（含部分未生成的章节）由用户决定是否处理
