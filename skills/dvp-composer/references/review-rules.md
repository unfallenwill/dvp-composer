# 逻辑审查规则

阶段四（逻辑审查）使用的校验规则定义。每条规则包含检查项、判断逻辑、严重级别。

## 严重级别定义

| 级别 | 标记 | 处理要求 |
|------|------|---------|
| 通过 | ✅ | 无需处理 |
| 警告 | ⚠️ | 用户决定是否修正 |
| 错误 | ❌ | 必须修正才能继续 |

---

## R1: 字段引用完整性

**检查内容**：编辑检查（Edit Checks）中引用的变量名是否在 CRF 字段列表中存在。

**判断逻辑**：
1. 提取所有编辑检查章节中 `变量` 或 `变量名` 列的值
2. 与 CRF 相关章节中的字段列表对比
3. 编辑检查中引用的变量不在 CRF 中 → ❌ 错误
4. CRF 中有字段但无对应编辑检查 → ⚠️ 警告（可能遗漏）

**触发条件**：DVP 中同时包含编辑检查章节和 CRF/数据域章节时执行。

---

## R2: 数据一致性

**检查内容**：同一条信息在不同章节中的值是否一致。

**判断逻辑**：
1. 提取 meta 区域和各章节中出现的共同字段（如研究名称、方案编号、访视名称）
2. 比较同一字段在不同位置出现的值
3. 值不一致 → ❌ 错误
4. 某处缺失但其他处存在 → ⚠️ 警告

**典型检查项**：
- 研究名称在 meta 和章节中是否一致
- 方案编号在 meta 和章节中是否一致
- 访视名称在不同章节间是否一致

---

## R3: 命名规范

**检查内容**：ID 编号的连续性和格式统一性。

**判断逻辑**：
1. 提取所有带 ID 前缀的标识符（如 EC001、CC001、VS001 等）
2. 按前缀分组检查：
   - 同类 ID 的前缀是否一致 → 不一致 ⚠️ 警告
   - 编号是否连续 → 不连续 ⚠️ 警告
   - 是否有重复 ID → ❌ 错误
3. 格式检查（如是否统一使用 3 位数字编号）

---

## R4: 业务逻辑合理性

**检查内容**：日期范围、数值区间、逻辑条件是否合理。

**判断逻辑**：
1. **日期检查**（根据字段语义模式匹配，不依赖固定字段名）：
   - 出生日期不能晚于研究日期 → ❌ 错误
   - 知情同意日期不能晚于首次给药日期 → ❌ 错误
   - 访视日期应在研究周期内 → ⚠️ 警告

2. **数值区间检查**：
   - 正常范围下限应小于上限 → ❌ 错误
   - 数值约束条件是否在合理范围内 → ⚠️ 警告

3. **逻辑条件检查**：
   - 条件表达式语法是否合理（如 `>` `<` `=` `!=` `in` `notin` 等操作符使用是否正确）
   - 空值条件 `!= ""` 和 `== ""` 是否矛盾 → ❌ 错误

---

## R5: 规则冲突检测

**检查内容**：同一字段是否存在矛盾的检查规则。

**判断逻辑**：
1. 按变量名分组所有编辑检查
2. 对同一变量的多条规则对比：
   - 一条要求必填（`!= ""`），另一条允许为空 → ❌ 错误
   - 一条设置上限，另一条设置的上限低于下限 → ❌ 错误
   - 两条规则检查相同条件但严重程度不同 → ⚠️ 警告

---

## R6: 交叉引用一致性

**检查内容**：章节间的 ID 引用是否对应。

**判断逻辑**：
1. 提取所有章节中出现的 ID 引用（如"参见 EC001"、"详见 CC003"等）
2. 验证被引用的 ID 是否存在
3. 引用的 ID 不存在 → ❌ 错误
4. 存在未在任何地方被引用的 ID → ⚠️ 警告

---

## 审查报告格式

```
=== DVP 逻辑审查报告 ===

R1 字段引用完整性
  ✅ CRF 字段共 45 个，编辑检查引用 38 个，全部在 CRF 中存在
  ⚠️ CRF 中有 7 个字段无对应编辑检查：[WEIGHT, HEIGHT, TEMP, PULSE, BP_SYS, BP_DIA, ECG]

R2 数据一致性
  ✅ 研究名称一致：ABC-123
  ✅ 方案编号一致：PROTO-2024-001
  ❌ 访视名称不一致："Screening" vs "筛选访视"（出现在不同章节中）

R3 命名规范
  ✅ 编辑检查 ID 前缀统一：EC
  ⚠️ 编号不连续：EC003 后缺少 EC004，直接跳到 EC005

R4 业务逻辑合理性
  ✅ 日期逻辑检查通过
  ✅ 数值区间检查通过

R5 规则冲突检测
  ❌ 变量 DOB 存在冲突规则：EC005 要求"必填"，EC012 允许"为空"

R6 交叉引用一致性
  ✅ 所有引用 ID 均存在

---
总计：✅ 8  ⚠️ 2  ❌ 2
错误项必须修正后才能生成输出。
```

## 审查执行方式

逻辑审查由 Claude 在交互过程中执行，基于已收集的 JSON 数据进行分析。不需要额外的脚本——Claude 根据 review-rules.md 中的规则逐项检查并输出报告。
